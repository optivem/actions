## Parameter findings ‚Äî External

#### check-commit-status-exists

- `input.sha` ‚Äî `name-mainstream-convention` per rubric ¬ß1's commit-SHA qualification rule (codified in rubric v4/v5, 2026-04-22). This is a flat-namespace input carrying a commit SHA; the rubric mandates `commit-sha` in flat namespaces. Description already treats it as a commit identifier ("typically an upstream-repo commit"). The adjacent `head-sha:` input is self-disambiguating via its `head-` qualifier, which exposes the gap: `sha:` alone leaves "which SHA?" to the description when the convention says qualify it.
  - **Proposed fix (Recommended ‚Äî best-practice):** rename `sha` ‚Üí `commit-sha` across `action.yml`, the internal `SUBJECT_SHA` env bridge (rename to `COMMIT_SHA` or keep, internal-only), and the 1 consumer call site in `gh-optivem/.github/workflows/gh-acceptance-stage.yml`.
  - Alternative: leave as-is and accept the rubric-mainstream drift. Lower-churn but leaves the flat-namespace ambiguity on an action that sits at the centre of the upstream-SHA bookkeeping pattern ‚Äî future callers who follow the mainstream `commit-sha` convention at their end will hit a surprise mismatch at this seam.

#### check-ghcr-packages-exist

- `output.exist` ‚Äî `output-name-verb-led` / `name-misleading`: singular `exist` is a bare verb form; every sibling `check-*-exists` action in the repo (`check-commit-status-exists.exists`, `check-tag-pattern-exists.exists`) returns `exists`. The mismatch is purely grammatical but it's the kind of detail consumers copy-paste incorrectly: reading the output as `exists` (common mental model) silently returns empty because the actual key is `exist`. Consumers currently read it as `steps.<id>.outputs.exist`, which is correct but inconsistent.
  - **Proposed fix (Recommended ‚Äî best-practice):** rename output `exist` ‚Üí `exists`. Update 18 consumer workflow files in `shop/.github/workflows/` (all `*-acceptance-stage*.yml` files) that read `steps.<id>.outputs.exist`.
  - Alternative: rename the sibling actions' `exists` ‚Üí `exist` for internal consistency on the singular spelling. Rejected ‚Äî mainstream convention (`test -e`, `ls`, `find`, most query APIs) uses `exists`, and the siblings already follow it. Moving siblings backwards would violate mainstream-first.

#### tag-docker-images

- `output.image-urls` ‚Äî `name-misleading`: this output carries the *retagged* image URLs and shadows the input `image-urls:` (which carries the *source* image URLs). A caller writing `steps.tag-images.outputs.image-urls` has to read the step id, then re-read the action's docs to know whether that's the input it passed in or the result it got back. The ambiguity is latent but real.
  - **Proposed fix (Recommended ‚Äî best-practice):** rename output `image-urls` ‚Üí `tagged-image-urls`. Update 24 consumer workflow files in `shop/.github/workflows/` (all `*-acceptance-stage*.yml` and `*-prod-stage*.yml` files across monolith/multitier √ó dotnet/java/typescript).
  - Alternative A ‚Äî rename output to `retagged-image-urls` (emphasises that the operation is server-side retag, not a fresh push). Slightly more honest about the mechanism; equally valid.
  - Alternative B ‚Äî leave as-is and rely on the description. Lower-churn but leaves the input/output shadowing in place on a 24-call-site action where the output is ALSO actively read (24/24 consumers consume it). The shadow is maximally exposed.

### Other

- **Fast-feedback sizing (¬ß1.3) ‚Äî `wait-for-endpoints` has no top-level `timeout-seconds` cap.** With defaults `max-attempts: 30` √ó `wait-seconds: 10` + exponential backoff capped at 16√ó (`MAX_BACKOFF=160s`), a fully-failing endpoint can wait up to roughly `sum_{i=1..30} min(10¬∑2^(i-1), 160)` ‚âà 4500s (75 min) per endpoint, with `N` endpoints running sequentially. The commit-stage budget in Farley's *Continuous Delivery* is ~5 minutes; acceptance-stage readiness probes commonly sit in the 5‚Äì15 minute band. 75 min/endpoint √ó N endpoints is an MTTR amplifier (¬ß1.3 DORA linkage) if a single service hangs. 87 call sites in `shop` ‚Äî this is the most-called action in the repo.
  - Options:
    1. **Recommended (best-practice):** add a top-level `timeout-seconds` input with a conservative default (e.g. `900` / 15 min) and wrap the polling loop in `timeout "$TIMEOUT_SECONDS"`. On timeout, dump the compose logs (already done on failure) and exit 124. Makes the cap visible to callers as an overridable knob. Classification: **External** if the default caps previously-unbounded behaviour below what some callers relied on ‚Äî but 15 min ceiling is unlikely to clip any healthy probe. Still classify as External for honesty (see tie-breaker rule).
    2. Lower-churn: document in the `description:` that callers should set `timeout-minutes:` at the job level. Internal-only (pure description edit). Leaves the default effectively unbounded and relies on caller discipline ‚Äî the rubric's own ¬ß1.3 language ("with a default that respects the target commit-stage budget") rejects this as the primary fix.
  - The recommendation is Option 1 because rubric ¬ß1.3 explicitly mandates "expose a `timeout-seconds` (or `max-attempts` + `poll-interval`) input with a default that respects the target commit-stage budget" ‚Äî the current `max-attempts` + `wait-seconds` nominally satisfies this but the exponential backoff cap pushes the real ceiling far beyond 5 minutes. Recommendation aligns the cap with the budget.
  - 87 call sites ‚Äî this would be the widest-reach external change in this audit. Co-ordinate with consumer updates.
