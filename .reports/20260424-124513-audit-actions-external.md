## Breaking changes

### Naming violations — External

| dir | rule violated | proposed name |
|---|---|---|
| `generate-release-notes` | rubric §4 `name-misleading` — the action's description states it "hardcodes the `🚀`/`PROD` title shape because every caller to date emits prod-deployed release notes", and internally composes a literal `"🚀 $RELEASE_VERSION PROD"` title. The generic name promises any release-notes variant; the implementation is specifically PROD-deployment. A consumer that wanted QA or signoff notes would — per the action's own description — "write a sibling action", meaning the current name is already occupying the generic slot for a specialised implementation. | `generate-prod-release-notes` (preferred — honest about specialisation), OR parameterise the title prefix + stage label and keep the generic name (alternate). Recommended: **rename to `generate-prod-release-notes`**. Reason: the PROD hardcoding is structural (title shape, notes body assumptions, `Release Version: / Promoted From:` framing all assume a prerelease → release promotion); making it parameter-driven would require either (a) a richer title template input plus body-section toggles — signficant input surface growth — or (b) a separate sibling anyway for non-PROD variants, since the current name would then read awkwardly against a full template input. Consumer churn: 12 call sites in shop (all `monolith-*-prod-stage*.yml` / `multitier-*-prod-stage*.yml` + `_*-release-pipeline.yml`). |

No other directory-level name violations in this run. All other directory names pass the kebab-case, verb-first, Tier-3-`github`-prefix-when-ambiguous, and honest-verb tests in rubric §3–§4.

**Examined and rejected — naming:**

- `check-tag-exists` vs `validate-tag-exists` — the split looks asymmetric ("`check-*` non-assert + `validate-*` assert") but matches the Mainstream-first principle (`check-*` = boolean return / non-asserting; `validate-*` = assert-and-fail) and is honest about the side-effect-on-failure semantics. KEEP as-is.
- `check-ghcr-image-tag-exists` vs `check-ghcr-packages-exist` — both prefixed `ghcr` per Tier 3 ambiguity rule (§3.1), both `check-*-exist(s)` per the Mainstream-first `check-*` preference. Different surfaces (batched per-(image,tag) probe vs homogeneous tag-across-images probe) justify two actions; see the Consolidation subsection below. No rename recommended.
- `get-commit-status` — verb `get-*` is misleading because the action hard-fails when no matching status is found — that's `validate-*`-shaped behaviour. Reclassifying as `validate-commit-status` would fix the verb honesty per §4 but would collide with the paired `check-commit-status-exists` (which fails open to `exists=false`) and muddy the discoverable mental model ("which action do I use for reading a status that must already exist?"). The rubric-honest fix is not a rename but a behaviour relaxation: `get-commit-status` should return empty outputs instead of failing when no match, making `get-*` honest. Classified as a behavioural change, not a naming change — covered under Parameter findings / DevOps alignment where `get-commit-status` is reviewed. Kept as "examined and rejected" here so the absence of a naming rename is visible.
- `check-ghcr-packages-exist` (the `-packages-` segment) — considered `check-ghcr-images-exist` for consistency with the sibling `check-ghcr-image-tag-exists`. Rejected: the action operates at the GHCR "package" granularity (the GitHub Packages model treats each repo/image path as a "package" and each tag as a "version"); the name aligns with GitHub's own vocabulary even though the implementation uses OCI under the hood. No rename; mainstream-first principle favours the ecosystem noun.

### Parameter findings — External (breaking)

#### `deploy-docker-compose`
- `input.environment` — `required-default-contradiction` (mild). `required: true` on an input whose own description declares "Display-only … The action does not route based on this value." A display-only label that cannot be empty is plausible but surprising — today's shape raises a schema error on a missing caller value, without any behavioural justification for the hard requirement. Three resolutions; **recommended (best-practice):** relax to `required: false` with `default: ''` and keep the description. Non-breaking under rubric §1.1 semantics (relaxing `required: true` → `required: false` cannot break existing callers that already pass a value) — but classified **External non-breaking** per the scope split (signature surface changed visibly). See the Non-breaking Parameter findings subsection below for the actual plan item.
- `input.version` — same shape as `input.environment`, same three resolutions, same recommendation, same classification. Also filed under External non-breaking.

These two are mentioned here (Breaking) only so the reader sees the full parameter review per action; the actionable plan items live under Non-breaking below.

#### `generate-release-notes`
- Naming rename above cascades into parameter-space: if the action is renamed to `generate-prod-release-notes`, inputs and outputs stay (`prerelease-version`, `release-version`, `artifact-urls`; `title`, `notes-file`), but the semantic identity "what this action generates" is explicit. No separate parameter finding; bundled with the naming item.

#### `get-commit-status`
- `runs` hard-fails when no matching status is found. The verb `get-*` implies read-only with empty-on-miss (rubric §4 canonical-semantics table). Three resolutions:
  1. **Recommended (best-practice):** relax the failure path — return empty outputs (`description=''`, `state=''`, `target-url=''`) on no match, instead of `exit 1`. Callers currently depending on the failure (the single `gh-release-stage.yml` call site) must be updated to check for empty output and branch accordingly. This makes `get-*` honest per rubric §4 and makes the action symmetric with `check-commit-status-exists` (fail-open sibling) and `create-commit-status` (write sibling). Breaking because the error-path contract changes: a caller that relies on the step's failure to short-circuit the job now needs an explicit empty-output check.
  2. Rename to `validate-commit-status` and keep the failure semantics. Preserves today's contract at one call site; muddies the `check-*` / `get-*` / `validate-*` trio by adding a `validate-*` flavour that doesn't do the full "assert-and-fail on any input shape" motion (it only asserts presence; context and state filter semantics are not clearly "validation"). Demoted: less mainstream-aligned per the `check-*`/`get-*` distinction.
  3. Leave as-is. Sacrifices: the verb-vs-behaviour mismatch stays; a future reader sees `get-*` and expects empty-on-miss, gets a hard-fail. Not recommended.
- **Option 1 recommended.** Consumer impact: 1 call site (`gh-optivem/.github/workflows/gh-release-stage.yml`). Classified breaking. Category: `devops-alignment` (the verb-vs-behaviour mismatch is an observability / honesty issue more than a name issue).

### Duplicates — External

None. Examined and rejected:

- `check-ghcr-image-tag-exists` vs `check-ghcr-packages-exist` — same underlying operation (OCI HEAD `/manifests/{tag}` against `ghcr.io`) but different input/output shapes. Test (a): same side effect on same target? Yes — both emit registry HEAD requests against a known image path, neither writes anything. Test (b): can a caller swap one for the other without changing inputs or outputs? **No.** `check-ghcr-image-tag-exists` takes `[{key, image-url, tag}]` and emits per-(image,tag) results plus an `any-exist` aggregate; `check-ghcr-packages-exist` takes a newline image list + single shared `tag` and emits a single `exist` boolean. The aggregate `any-exist` from the batched action does overlap with `exist` from the simpler action, but the input shape doesn't. Not a duplicate. Covered instead under Consolidation — see "Shared GHCR OCI probe helper" in the Internal report.
- `wait-for-github-workflow` vs `trigger-and-wait-for-github-workflow` — the second is a strict superset of the first's watch phase + a dispatch step. Same side effect on same target? No — one triggers, one only observes. Test (b): cannot swap. Not a duplicate. Shared watch logic is extractable but is intentionally inlined (both actions already source `shared/gh-retry.sh`). See Internal.
- `read-base-version` vs `read-base-versions` — singular vs plural (batched keyed form). Different surface entirely (single file vs JSON array of keyed entries). Not a duplicate; both are called.
- `check-commit-status-exists` vs `get-commit-status` — same subject (commit statuses on a SHA) but different polarities: one is a fail-open boolean probe, the other is a fail-closed single-match read that emits `description` / `state` / `target-url`. Not a duplicate — the output surfaces differ, and the failure policy differs.
- `validate-tag-exists` vs `check-tag-exists` — ship deliberately as a paired read/assert (inherited from the symmetry refactor applied since the prior audit — see Prior-plan reconciliation). Same input shape, opposite failure semantics. Not a duplicate; both are first-class.
- `compose-docker-image-urls` vs `compose-tags` — both are string-transform "apply a tag to a base" primitives, but `compose-docker-image-urls` is specialised to Docker image URL shape (accepts JSON or newline list, outputs JSON), while `compose-tags` is generic keyed-version templating (JSON `[{key, version}]` → JSON `[{key, tag}]` with a caller-supplied template). Not a duplicate — the keyed form preserves caller-supplied identifiers across the transform, the specialised form doesn't. Consolidation opportunity was considered and rejected: `compose-docker-image-urls` could theoretically be expressed as `compose-tags` with a specific template, but the input shapes diverge (raw JSON array of strings vs keyed JSON of objects), and the caller-visible output shape diverges (JSON array of strings vs JSON `[{key, tag}]`). Merging would force all 24 shop callers into the keyed form.
