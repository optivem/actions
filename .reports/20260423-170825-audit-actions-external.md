### Consolidation opportunities — External

#### `validate-tag-exists` / `validate-version-unreleased` symmetry (user-flagged focus area)

**Practice under review.** Whether the pair `validate-tag-exists` (assert tag DOES exist) and `validate-version-unreleased` (assert tag does NOT exist) should be reshaped toward greater symmetry — shared input vocabulary, shared output shape, possibly a common internal helper or consolidation with a mode flag.

**What's asymmetric today.** (Cross-referenced with `actions/plans/2026-04-23-validate-tag-pair-symmetry.md`, the user's parked idea file.)

| Dimension | `validate-tag-exists` | `validate-version-unreleased` |
|---|---|---|
| Input name | `tag` | `version` (but value is a git tag) |
| Impl | `git ls-remote --tags` (remote) | `git fetch --tags` + `git tag -l` (local) |
| Cross-repo capable | yes (`repository`, `token`, `git-host`) | no — locked to checkout |
| Skip mode | none — always fails | `on-already-released: fail\|skip` |
| Output | none | `already-released` (only when skip mode) |
| Error message | generic | release-domain ("Bump VERSION…") + optional workflow links |

**Evaluated options.**

1. **Option 1 — Recommended (best-practice).** Reshape into symmetric pair. Three coordinated changes:
   - Rename `validate-version-unreleased` → `validate-tag-not-exists` (directory + `name:`).
   - Rename input `version` → `tag`.
   - Port implementation from local `git tag -l` to remote `git ls-remote --tags`; add `repository` / `token` / `git-host` inputs (matching `validate-tag-exists`). Net cost to the action is zero — the existing `git fetch --tags` is already a remote round-trip.
   - Generalise the skip semantics: `validate-tag-exists` gains `on-missing: fail|skip` + `missing` output; `validate-tag-not-exists` renames `on-already-released` → `on-found`, `already-released` → `found`. Preserve today's release-pipeline error text behind a new optional input `recovery-workflow-files:` (JSON object `{bump, rc}`) or split into a sibling release-flavoured wrapper action if the error-text coupling matters.

   Rationale — this is the **rubric-aligned best-practice option** per the Mainstream-first principle + rubric §4 (honest verb/subject), §3.2 (prefer VCS, works already), and §1.5 vocabulary (input name describes the shape carried, which is `tag`). It lines up the pair as symmetric inverses (both `validate-tag-*`), puts the shared machinery in one repo-internal helper (see the Internal-report duplicate finding), and makes the "not-exists" action cross-repo capable for the first time, which is the exact gap the user flagged as a latent feature request.

   Consumer churn: ~37–60 call sites across shop + gh-optivem — 30 shop files × 1–2 call sites for validate-tag-exists (output `missing` opt-in only when callers adopt `on-missing: skip`), 19 shop files + 1 gh-optivem file for validate-version-unreleased (directory + input rename; output rename where `already-released` is read — ~6 acceptance-stage files).

   **Pedagogical impact (teaching-clarity override check).** The review at `actions/plans/2026-04-23-validate-tag-pair-symmetry.md` notes 3 lesson files under `courses/01-pipeline/accelerator/course/` depend on the current names: `04-qa-stage/01-deploy-to-qa.md`, `04-qa-stage/04-qa-signoff.md`, `05-production-stage/01-validate-qa-signoff.md`. Per rubric §2 (teaching-clarity override — real-world best practice wins), the rubric-aligned consolidation still wins but the lesson files must be updated alongside the code change. Call that out explicitly in any execution plan.

2. **Option 2 — Name-only rename.** Rename directory + input only; keep the implementation asymmetric (local vs remote), keep `on-already-released` / `already-released` vocabulary, keep the domain-flavoured error message.
   - Lower churn: ~20 call sites (directory + input) + 6 output-rename-dependent sites if `already-released` is renamed too (recommend keeping it under this option).
   - Sacrifices: the cross-repo capability stays off; the "I want to assert a tag does not exist on optivem/shop" request still has no action to satisfy it; the rubric §3.2 "prefer VCS over platform API" and §1.5 vocabulary gaps stay open.
   - This is the fallback option the user's parked note recommends as "middle ground." Leaves the structural symmetry gap but fixes the discoverability gap (the not-exists action is now findable by name).

3. **Option 3 — Defer (status quo).** Leave both as-is.
   - Zero churn. Sacrifices: everything above, plus a continuing risk that the third and fourth `validate-*` actions are added before the convention lands, cementing the asymmetry.

4. **Option 4 — Merge into one `validate-tag` with a `mode: must-exist|must-not-exist` flag.** (Considered per the user's explicit prompt to evaluate consolidation with a mode flag.)
   - Surface: one action, `mode: must-exist` (default) or `must-not-exist`; shared `tag`, `repository`, `token`, `git-host`, `on-missing`/`on-found` inputs unified into `on-violation: fail|skip` + a polymorphic output `violation` (boolean) + `tag-found` (boolean echo).
   - **Rejected.** Two reasons. First, rubric §6 one-concern-per-action: "asserting tag must exist" and "asserting tag must NOT exist" are opposite CI gates, not two projections of a single atomic operation (§6.3 does not apply — each path computes and acts on the same boolean but the *decision* each gate enforces is different; the payoff of splitting is swappability of the *policy*, which is exactly what a two-action pair gives you). Second, call-site readability: `uses: validate-tag-exists` vs `uses: validate-tag-not-exists` reads as an immediate boolean assertion; `uses: validate-tag` with `mode: must-not-exist` adds a cognitive hop every reader of the workflow pays on every call. With 44 + 20 call sites, that's a lot of added noise for no benefit. Flagged explicitly per the rubric §2 teaching-clarity + mainstream-first lens: the two-action shape is the mainstream GitHub Actions pattern (compare `actions/cache` + `actions/cache/restore`, `docker/login-action` + `docker/logout-action` — the ecosystem prefers two side-effecting verbs over one with a mode).

**Recommendation: Option 1 (full symmetry refactor).** Best-practice-aligned per rubric §4/§3.2/§1.5 and closes the real feature gap (cross-repo "not exists"). Option 2 is the acceptable lower-churn fallback if capacity is constrained. Option 4 (mode-flag merge) is explicitly rejected. Option 3 (defer) stays on the table only if no other touch is happening in the release pipeline this quarter; the user's parked note already recommends deferring in the absence of a trigger, and this audit does not itself constitute a trigger — it is the finding.

Tagged as breaking (sketch of consolidated surface):

```yaml

  tag: { required: true }
  on-missing: { required: false, default: 'fail' }   # fail | skip

# validate-tag-not-exists (formerly validate-version-unreleased)
inputs:
```
