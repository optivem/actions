## DevOps alignment findings — Internal

- **Tool-agnostic composition / Prefer VCS over platform API** — `publish-tag/action.yml` lines 50 and 72 use `git ls-remote --tags origin "refs/tags/$TAG"` rather than `git ls-remote --tags "$push_target" "refs/tags/$TAG"` (or, more correctly, the read-only counterpart `remote_query_url`). The push path uses the parameterised target correctly (line 67), so the inconsistency is internal — for the *current caller set* where `repository` defaults to `github.repository` and the local checkout matches, the result is identical. But the action accepts `repository` and `git-host` inputs; a caller wiring up a non-default `repository` would silently get queries against whatever `origin` happens to point at on the runner (typically the calling repo, not the target). Same shape as the (now-deleted) `create-component-tags` portability bug. **Internal preservative refactor:**
  - **Recommended (best-practice).** Replace the two `git ls-remote --tags origin` invocations on lines 50 and 72 with `git ls-remote --tags "$push_target" "refs/tags/$TAG"` (already in scope) — re-uses the same auth-bearing URL the push uses, no new variables. Behaviour-preserving for any caller passing `repository == github.repository` (today's reality across all 26 call sites). Categorisation: `devops-alignment` (portability). Consumer impact: none; descriptions and surface unchanged.
  - **Alternative — derive a separate read-only URL via `remote_query_url`.** Sacrifices: two URL composers in the same action; slightly more code. Lower-effort but adds variable.
  - **Option 1 recommended.** Filed in plan as the only internal item this run that is observably a portability bug rather than cosmetic.

  Note: this finding existed in the 2026-04-28 audit's `create-component-tags` block (now obsolete) but the same pattern in `publish-tag` was not flagged then. Re-examining `publish-tag` from scratch this run: bug present, but only observable via a non-default `repository` input. Filed Internal because no current caller exercises the buggy path.

- **Tool-agnostic composition — `evaluate-run-gate` JSON shape.** The `skip-conditions` input description names the `${ expression }` pre-resolution shape: callers must pre-render `when:` to a literal `"true"`/`"false"` string before passing the JSON. Internal style observation: this pre-resolution is opaque to a reader who doesn't read the description, and the pattern depends on GitHub-Actions-specific expression syntax. Considered flagging as Tier 3 leakage — rejected because the action itself does no expression evaluation (it consumes pre-resolved booleans), so it is portable in principle (a GitLab port would pre-resolve in YAML differently). Action-name remains correct (Tier 1). No finding.
