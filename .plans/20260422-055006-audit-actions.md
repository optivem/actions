- [ ] **`trigger-and-wait-for-github-workflow` hardcodes a 10-second `sleep` between trigger and run-ID lookup** — rubric §1.3 "Bounded retry with backoff": this is a flat sleep, not retry — so not a retry-budget violation. But `gh run list --limit 1` may race and pick up a run from a *previous* triggering if the dispatch is slow. Action does not verify the matched run's `headSha`. Filing as a correctness concern, not a naming concern. Consider polling `gh run list` until a run with matching `headSha` (or created_at > dispatch time) appears instead of a fixed 10s sleep.
  - Affects: `trigger-and-wait-for-github-workflow`
  - Consumers to update: none (internal behavior change)
  - Category: devops-alignment

VJ: what's the suggestion?

**Agent:** Replace the 10s flat `sleep` + `--limit 1` lookup with bounded polling that matches on **createdAt timestamp** (since `workflow_dispatch` doesn't change HEAD, matching on `headSha` alone isn't sufficient — two back-to-back dispatches on the same ref would both have the same `headSha`).

Concrete diff sketch for `trigger-and-wait-for-github-workflow/action.yml`:

```bash
# Before triggering, capture the boundary timestamp
dispatch_time=$(date -u +%Y-%m-%dT%H:%M:%SZ)

# Trigger workflow (unchanged)
gh_retry workflow run "$WORKFLOW" --repo "$REPOSITORY" --ref "$REF" $input_flags
echo "Triggered $WORKFLOW at $dispatch_time"

# Poll for a run created after dispatch_time
max_wait=60
elapsed=0
run_id=""
while [ "$elapsed" -lt "$max_wait" ]; do
  sleep 5
  elapsed=$((elapsed + 5))
  run_id=$(gh_retry run list \
    --workflow="$WORKFLOW" \
    --repo "$REPOSITORY" \
    --branch "$REF" \
    --limit 5 \
    --json databaseId,createdAt,headSha \
    --jq "[.[] | select(.createdAt >= \"$dispatch_time\")] | .[0].databaseId")
  if [ -n "$run_id" ] && [ "$run_id" != "null" ]; then
    break
  fi
done

if [ -z "$run_id" ] || [ "$run_id" = "null" ]; then
  echo "::error::Dispatched $WORKFLOW but no run appeared within ${max_wait}s"
  exit 1
fi

echo "run_id=$run_id" >> "$GITHUB_OUTPUT"
```

Key changes: (1) capture `dispatch_time` before trigger, (2) poll every 5s up to 60s, (3) filter by `createdAt >= dispatch_time`, (4) fail with a clear error if nothing appears.

No input changes, no consumer updates. Approve?
