#### bump-patch-versions

## DevOps alignment findings — External

### Other

- **Fast-feedback sizing (§1.3)** — `trigger-and-wait-for-github-workflow` has no hard timeout ceiling. `gh run watch --exit-status` blocks until the triggered run terminates; an upstream stuck run can wedge the caller indefinitely.
  - Options:
    1. **Recommended (best-practice):** add a top-level `timeout-seconds` input with a conservative default (e.g. `1800` / 30 min). Wrap `gh run watch` with a deadline loop that kills the watch and surfaces `status=timed-out` as an output (dual-surface observability per §1.4).
    2. Lower-churn: document that the caller workflow should set `timeout-minutes:` at the job level — does not expose the timeout as a first-class input.

- **Fast-feedback sizing (§1.3)** — `wait-for-github-workflow` has unbounded total wait. `max-discovery-attempts: 120` × `poll-interval: 30s` = 60 min of discovery alone, then unbounded `gh run watch`. Same remedy: add a hard `timeout-seconds` input.

- **`compose-release-notes` spans multiple concerns.** Per rubric §6: the action (a) composes a title string, (b) writes a markdown notes file, (c) conditionally appends an artifacts section. The name `compose-*` promises a pure data transform but the action has a filesystem side effect (mktemp file). Recommended: rename to `generate-release-notes` (see Naming violations) AND keep both outputs — splitting into `compose-release-title` + `generate-release-notes-file` has not proven necessary at current call-site scale.
