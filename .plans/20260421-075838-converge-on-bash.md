# Converge on bash — retire PowerShell in `optivem/actions` — 2026-04-21 07:58 UTC

## Motivation

The `optivem/actions` repo currently mixes two shells:

- **35** `action.yml` files use `shell: bash` (~42 step occurrences)
- **11** `action.yml` files use `shell: pwsh` (~13 step occurrences)
- **~10** non-test `.ps1` scripts (plus test harnesses)

The split is already ~80/20 bash-dominant. Every cross-cutting concern pays the tax twice — the most recent example is the [gh-retry plan](20260420-154500-gh-retry-transient-errors.md), which has to build two helpers ([Invoke-GhWithRetry.ps1](../shared/Invoke-GhWithRetry.ps1) + [gh-retry.sh](../shared/gh-retry.sh)), two self-tests, and a lint rule that understands both. Future concerns (structured logging, rate-limit handling, auth) will repeat this.

All actions run on GitHub-hosted Linux runners, so pwsh's cross-platform/Windows story buys nothing here. Bash + `jq` covers every capability the PS1 scripts currently use (JSON munging, `gh` calls, string work).

## Principle

**New actions are bash-only. Existing PowerShell is migrated systematically — port every `.ps1` / `shell: pwsh` step to bash unless there is a concrete reason not to.** If a specific item should *not* be ported (e.g. the logic genuinely depends on a Windows-only cmdlet, a .NET object pipeline that bash+jq can't reasonably replicate, or the regression risk outweighs the benefit), **skip that item and raise it to the human** with a one-line rationale — do not silently leave it behind and do not force the port. The goal is full convergence; exceptions are explicit and documented, not implicit and forgotten.

## Target state

- `shell: pwsh` does not appear in any new `action.yml`.
- No new `.ps1` files are added under `optivem/actions` (except as transitional test harnesses).
- [README.md](../README.md) documents the bash-only policy and lists any explicitly-skipped exceptions with their rationale.
- A lint check fails PRs that introduce new `shell: pwsh` or new `*.ps1` files.

## Items

### Phase 2 — systematic migration

Port every item below to bash. Work through the list in the order shown (ascending complexity — banks confidence with the small ones before tackling the gnarly `gh`-heavy scripts). For each item:

1. Port the logic to bash (using `jq` for JSON, `gh_retry` from [shared/gh-retry.sh](../shared/gh-retry.sh) for `gh` calls).
2. Re-point the `action.yml` to `shell: bash`.
3. Delete the `.ps1` file(s).
4. Verify via the Phase 4 runtime check before moving on.

**Skip-and-flag rule:** If during porting you conclude a specific script genuinely should *not* be bash (e.g. relies on .NET object semantics that bash+jq can't cleanly replicate, touches Windows-only behavior, or the port would meaningfully increase regression risk on a high-blast-radius action), **stop, leave the file as-is, and raise it to the human** with a one-line rationale. Do not force the port. Do not silently skip. The human decides whether to keep it as a documented exception or push through.

**Small (single-purpose PS1, mostly string/arg handling):**

- [ ] `tag-docker-images/Tag-DockerImages.ps1` + `tag-docker-images/action.yml`
- [ ] `resolve-docker-image-digests/action.ps1` + `resolve-docker-image-digests/action.yml`

**Medium (some logic, JSON handling):**

Any item skipped under the skip-and-flag rule above must be recorded in [README.md](../README.md) as an explicit exception (one line: path + reason), so future readers don't mistake it for "we just haven't gotten to it yet."

## Open decisions

- **Should Phase 1's lint rule allow `shell: pwsh` inside existing directories that still have pwsh steps, to avoid blocking unrelated edits during the Phase 2 migration window?** **Recommended: yes** — the rule should only flag *new directories* and *new `.ps1` files* until Phase 2 completes; once the backlog (minus any skip-and-flagged exceptions) is drained, tighten the lint to reject any `shell: pwsh` or `.ps1` outside the documented exception list.
- **Dev-time test scripts** (`test-both-formats.ps1`, `comprehensive-test.ps1`, etc. under `resolve-docker-images/`) — port to `.sh` or just delete? **Recommended: delete** if they're not run in CI and the main script has coverage elsewhere; port only if they document a real test case.
