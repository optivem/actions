# Converge on bash — retire PowerShell in `optivem/actions` — 2026-04-21 07:58 UTC

## Motivation

The `optivem/actions` repo currently mixes two shells:

- **35** `action.yml` files use `shell: bash` (~42 step occurrences)
- **11** `action.yml` files use `shell: pwsh` (~13 step occurrences)
- **~10** non-test `.ps1` scripts (plus test harnesses)

The split is already ~80/20 bash-dominant. Every cross-cutting concern pays the tax twice — the most recent example is the [gh-retry plan](20260420-154500-gh-retry-transient-errors.md), which has to build two helpers ([Invoke-GhWithRetry.ps1](../shared/Invoke-GhWithRetry.ps1) + [gh-retry.sh](../shared/gh-retry.sh)), two self-tests, and a lint rule that understands both. Future concerns (structured logging, rate-limit handling, auth) will repeat this.

All actions run on GitHub-hosted Linux runners, so pwsh's cross-platform/Windows story buys nothing here. Bash + `jq` covers every capability the PS1 scripts currently use (JSON munging, `gh` calls, string work).

## Principle

**New actions are bash-only. Existing PowerShell is migrated opportunistically — only when the file is already being touched for another reason.** No big-bang rewrite. Over 6–12 months the pwsh surface area naturally shrinks to zero and the duplicate helpers can be deleted.

## Target state

- `shell: pwsh` does not appear in any new `action.yml`.
- No new `.ps1` files are added under `optivem/actions` (except as transitional test harnesses).
- [shared/Invoke-GhWithRetry.ps1](../shared/Invoke-GhWithRetry.ps1) and [shared/_test-gh-retry.ps1](../shared/_test-gh-retry.ps1) are deletable once the last PS1 consumer is gone.
- [README.md](../README.md) documents the bash-only policy and the opportunistic-migration rule.
- A lint check fails PRs that introduce new `shell: pwsh` or new `*.ps1` files.

### Phase 2 — opportunistic migration rule

No work items here — this phase is a **standing rule**, not a scheduled task:

> When you open any of the files below for another reason (bug fix, feature change, retry-wrapper migration, etc.), also port the surrounding `pwsh` step or `.ps1` script to bash **in the same PR** unless doing so would triple the diff. If the port is too large, file a dedicated migration item and keep the PR focused.

## Open decisions

- **Should Phase 1's lint rule allow `shell: pwsh` inside existing directories that still have pwsh steps, to avoid blocking unrelated edits?** **Recommended: yes** — the rule should only flag *new directories* and *new `.ps1` files*. Existing pwsh stays untouched until opportunistically migrated. One-line allowlist based on `git diff --name-only origin/main --diff-filter=A`.
- **Dev-time test scripts** (`test-both-formats.ps1`, `comprehensive-test.ps1`, etc. under `resolve-docker-images/`) — port to `.sh` or just delete? **Recommended: delete** if they're not run in CI and the main script has coverage elsewhere; port only if they document a real test case.
- **Relationship to the in-flight retry plan**: do phase 2 of the retry plan (migrate 5 `.ps1` scripts to use `Invoke-GhWithRetry`) **before** starting this plan's migrations, so we don't rewrite twice. Each bash port then uses `gh_retry` from the start. **Recommended: yes, retry plan first.**
