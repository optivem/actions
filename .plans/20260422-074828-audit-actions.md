- [ ] **`compose-prerelease-version` rejects `prefix` values containing a hyphen but doesn't document the constraint (§1.5 SemVer validation)** — the composer output shape is `{prefix}-v{base}-{suffix}.{build-number}`. A prefix like `monolith-java` (three hyphens including the join) still produces a valid SemVer pre-release (`monolith-java-v1.0.0-rc.7` — the pre-release segment is `rc.7`), but a consumer inspecting the string by splitting on `-` would trip. Flag as `type-shape-ambiguous`: add to the `prefix` description a worked example showing the emitted tag shape and a reminder that the caller's downstream parsers must split on `-v` / `-` with awareness, not blind split on first hyphen.

- [ ] **`compose-release-version` strip regex is case-insensitive and locked to `rc|alpha|beta|preview` — hardcoded set without matching `compose-prerelease-version` contract** — the `sed -E` regex `s/-(rc|alpha|beta|preview)\.[0-9]+$//I` silently strips only these four suffixes. Any other SemVer-§9-valid suffix (e.g. `dev`, `nightly`, `hotfix`) passes through unchanged, meaning the "release version" still carries a prerelease segment. This is a correctness gap between the two composer actions. Recommended: (1) narrow the documented `suffix` set in `compose-prerelease-version` to match this regex exactly, AND (2) fail-fast in `compose-release-version` when the stripped output still matches `-[A-Za-z0-9-]+\.[0-9]+$`. Better than silent pass-through.
  - Affects: `compose-release-version`, `compose-prerelease-version`
  - Consumers to update: none (tightens validation; all current consumers use `rc` suffix)
  - Category: devops-alignment
