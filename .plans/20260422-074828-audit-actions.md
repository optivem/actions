Per rubric §1.6 (Container image build/tag/push — use mainstream composite actions) this is the three-primitive split that the rubric explicitly rejects for Docker: it forces every image through a local-daemon round trip, skips BuildKit's remote cache and multi-arch handling, and emits no attestation. - [ ] **`compose-prerelease-version` rejects `prefix` values containing a hyphen but doesn't document the constraint (§1.5 SemVer validation)** — the composer output shape is `{prefix}-v{base}-{suffix}.{build-number}`. A prefix like `monolith-java` (three hyphens including the join) still produces a valid SemVer pre-release (`monolith-java-v1.0.0-rc.7` — the pre-release segment is `rc.7`), but a consumer inspecting the string by splitting on `-` would trip. Flag as `type-shape-ambiguous`: add to the `prefix` description a worked example showing the emitted tag shape and a reminder that the caller's downstream parsers must split on `-v` / `-` with awareness, not blind split on first hyphen.
  - Affects: `compose-prerelease-version`
  - Consumers to update: none (description-only)
  - Category: parameter-description

- [ ] **`compose-release-version` strip regex is case-insensitive and locked to `rc|alpha|beta|preview` — hardcoded set without matching `compose-prerelease-version` contract** — the `sed -E` regex `s/-(rc|alpha|beta|preview)\.[0-9]+$//I` silently strips only these four suffixes. Any other SemVer-§9-valid suffix (e.g. `dev`, `nightly`, `hotfix`) passes through unchanged, meaning the "release version" still carries a prerelease segment. This is a correctness gap between the two composer actions. Recommended: (1) narrow the documented `suffix` set in `compose-prerelease-version` to match this regex exactly, AND (2) fail-fast in `compose-release-version` when the stripped output still matches `-[A-Za-z0-9-]+\.[0-9]+$`. Better than silent pass-through.
  - Affects: `compose-release-version`, `compose-prerelease-version`
  - Consumers to update: none (tightens validation; all current consumers use `rc` suffix)
  - Category: devops-alignment

- [ ] **`check-timestamp-newer` relies on `string > string` bash lexicographic comparison on ISO 8601 timestamps (§1.3 fail-fast correctness)** — `if [[ "$SUBJECT" > "$BASELINE" ]]`. This works ONLY when both strings are UTC Z-suffixed same-format. If a caller passes a timezone-aware timestamp (e.g. `2026-04-22T10:15:00+00:00`) or one with different fractional-second precision, the comparison silently mis-orders and the workflow drops runs it should execute (or runs it should skip). Replace the bash comparison with explicit `date -d` normalisation before comparing, OR fail-fast if either input doesn't match the documented `YYYY-MM-DDTHH:MM:SSZ` shape.
  - Affects: `check-timestamp-newer`
  - Consumers to update: none (internal correctness fix)
  - Category: devops-alignment

- [ ] **`check-tag-pattern-exists` and `resolve-commit` accept `token` input but use it via direct URL-embedding (§1.4 Secrets / auth)** — both actions build `https://x-access-token:${TOKEN}@${GIT_HOST}/${REPO}.git` and hand that to `git ls-remote` / `git fetch`. The TOKEN ends up rendered into the literal command line, which violates the §1.4 guidance "Never interpolate secrets into a rendered command". Correct pattern: use `git -c http.extraheader="Authorization: bearer $TOKEN" fetch ...` with `TOKEN` set via step-level `env:` only. Rubric §3.3 specifically carves out the URL-embedding pattern as acceptable for Tier 2 portability today, but the §1.4 secret-masking concern still applies. Recommended: migrate to the `http.extraheader` pattern while keeping the `token` input shape unchanged.
  - Affects: `check-tag-pattern-exists`, `ensure-tag-exists`, `publish-tag`, `create-component-tags`, `resolve-commit`, `resolve-latest-prerelease-tag`, `resolve-latest-tag-from-sha`
  - Consumers to update: none (internal)
  - Category: devops-alignment
