### Parameter findings — External (breaking)

#### `resolve-latest-deployed-prerelease`

- **`output.ref` description vs runtime polarity mismatch.** Description: `'The ref of the latest successful deployment matching the criteria. Empty when none found (in which case the action also fails the step).'` But the `runs:` block **always** `exit 1`s on no-match (line 68–69) — the empty-when-none-found clause is unreachable. Consumers reading the description will reasonably model this as *empty-output OR fail*, when in practice it is always *fail*. Two resolutions:
  1. **Recommended (best-practice — fix the contract honestly).** Change the description to: `'The ref of the latest successful deployment matching the criteria. The action fails the step (exit 1) when none found — `ref` is never empty in practice.'` Description-only — but flagged here as breaking because **callers may have written defensive `if: steps.x.outputs.ref != ''` guards based on the prior contract**, and clarifying the contract may invalidate those guards. Categorisation: `parameter-deprecation`-adjacent (contract clarification, not deprecation per se). Consumer impact: requires a one-line scan of the 6 shop call sites to verify no caller is branching on empty.
  2. **Alternative — make the description match by relaxing the runtime to return empty.** Change the `runs:` block to write `ref=` then `exit 0` instead of `exit 1` when no match. Sacrifices: the resolver becomes a non-asserting read (semantically `find-*`/`get-*`), which would also force renaming the action to match (rubric §4 — `resolve-*` is read-only but doesn't carry "must succeed"; the current "fail on no match" semantics are closer to `validate-latest-deployed-prerelease`). The cascade is larger than option 1.
  - **Option 1 recommended.** Categorisation: `parameter-deprecation`. Classify breaking by the external-file tie-breaker — the contract is observably visible to callers.

- **Consider dropping the `repository` input.** `runs:` queries `repos/${REPOSITORY}/...` so the input *is* honoured for queries (unlike the deleted `create-component-tags`'s portability bug). However, the action does NOT accept a `git-host` input — it is hardcoded against the GitHub Deployments API. As a Tier 3 action whose entire premise is GitHub-API-bound, this is correct (`git-host` would be meaningless for a Deployments API query). **No finding** — calling out for transparency since two of the new actions inherit the Tier 3 lock-in cleanly while two (the resolvers) might tempt a future maintainer to add `git-host`. Reject that path: GitHub Deployments and GitHub Check Runs do not have a portable VCS analogue.

#### `create-check-run`

- **`input.name` collides with the kebab-case-action-name namespace.** The input is `name:`. Callers reading a `with: { name: qa/signoff, ... }` block alongside `with: { check-name: ... }` on the sibling `check-run-exists` will silently miss the asymmetry — the sibling uses `check-name`, this one uses bare `name`. Two resolutions:
  1. **Recommended (best-practice — rename for sibling-pair symmetry).** Rename `input.name` → `input.check-name`. The sibling primitives (`check-run-exists.check-name`, `resolve-latest-prerelease-with-check[-run].check-name`) all use `check-name`; this writer should match. Categorisation: `parameter-naming` (`name-mainstream-convention` — within-repo sibling-pair convention is the relevant mainstream here). Consumer churn: 6 shop call sites must update `name:` → `check-name:`.
  2. **Lower-effort — KEEP as `name`.** Sacrifices: leaves the asymmetry in place; readers paging through the trio see two `check-name` inputs and one `name` input and have to remember which is which.
  - **Option 1 recommended.** Categorisation: `parameter-naming`. Classify breaking — input rename is always breaking when external.

- **`input.title` default `''` semantics drift from sibling defaults.** When `title` is empty, the action coerces to `$NAME` (lines 57). The sibling `create-commit-status` has `description:` default `''` with no coercion. The pattern is fine on its own — but the description should say "Empty -> uses the check run name (after any rename of `name` to `check-name`, this clause cascades)." Already does in current copy, just flagging that the rename in the prior bullet propagates here. **Not a separate finding.**

#### `create-deployment`

- **`input.description` and `input.state` are paired with the `description`/`state` fields of the GitHub Deployments API**, but the *deployment record* and the *deployment status record* are two different API resources, and the action writes the same `description` to both (line 49 + line 61) and the same `state`-derived field to the status only. The current input shape conflates "describe the deployment" and "describe the most recent status update" — fine for this action's use case (one immediate status-write following the deployment), but a future caller wanting a different description on the status update has no surface for it. Two resolutions:
  1. **Recommended (best-practice if the action is to remain a one-shot writer).** Document the conflation explicitly in the description: `'Human-readable description recorded on **both** the deployment record AND its initial status update. Empty -> defaults to "Deploy of <ref>".'` Description-only, classify Internal (preservative). Filed in **Parameter findings — Internal** below, not here.
  2. **Alternative — split into `description` and `status-description`.** Adds one input; the missing input would default to the existing `description` value. Speculative — not recommended without a caller who needs the split (rubric §1.1 YAGNI).
  - **Option 1 recommended.** Move to Internal. Not a Breaking finding here.

#### `resolve-latest-prerelease-with-check`

- **`output.base-tag` is a duplicate of `output.tag` by construction.** The description acknowledges this explicitly: `'Alias of \`tag\`. Provided for parity with resolve-latest-prerelease-tag, which exposes both tag (with optional suffix) and base-tag (suffix stripped). Since rc tags here have no suffix, they are equal.'` Two resolutions:
  1. **Recommended (best-practice — drop the duplicate output).** Rubric §1.2 guidance is "default to keep" for outputs, but the threshold to KEEP is "primary return value" or "observability metadata." `base-tag` here is neither — it is parity-with-sibling that documents itself as redundant. Drop. Categorisation: `dead-output`. Classify breaking. Consumer churn: any caller currently reading `outputs.base-tag` must switch to `outputs.tag` (a search-and-replace).
  2. **Lower-effort — KEEP.** Sacrifices: surface clutter; readers see two outputs that are guaranteed equal.
  - **Option 1 recommended.** Categorisation: `dead-output`. The §1.2 default-to-keep does not apply because the output explicitly self-documents as redundant (no observability gain over `tag`).
